非常精實的介紹 clickhouse 文章。

ClickHouse 以 column-oriented 儲存的 database，主要用於 OLAP (online analytical processing) 線上分析處理的查詢。

表引擎的種類

clickhouse 可以建立多個節點來支援大量的數據計算，也會分成不同種類的表引擎來處理不同的資料特性。

單節點表引擎

這些引擎負責數據的物理儲存、排序、壓縮和過濾。它們只管「自己腳下」的那塊硬碟，不具備跨機器溝通的能力。

原理與核心概念請參考文章

Replacing 特性

依據 order by 去除重複，合併去重、分區去重。由於不知道底層什麼時候才會去重，因此如果確保資料的準確性與一致性的話，不建議使用。但可以在 select 之前加強制執行的語法。

高可用引擎(Replicated 系列)

屬於單節點表引擎，但具備「多機同步」能力。 將寫入動作同步給同一個 Shard 的其他 Replica。但它不會幫你做「分片（Sharding）」。它只保證這台機器跟它的備援機器的資料是一模一樣的。

Special 引擎

不存數據，像是一個窗口，負責把你的指令轉發到正確的地方。

  • Distributed（分散式引擎）：它就像是一個路由器。當你查它時，它去問各個節點上的 Local 表；當你寫它時，它把數據分發給各個 Local 表。
  • View / MaterializedView（視圖）：負責邏輯轉換或觸發數據處理。

其他

  • 刪除修改等動作對 clickhouse 來說效能負擔較重，因此可以透過打 tag 的方式過濾 tag 來組成資料最新狀態是比較好的做法。
  • clickhouse 沒有 master、worker 之分，機器透過 zookeeper 互爲副本

Sorting Key 是什麼？

在 clickhouse 中，Primary Key = Sorting Key = ORDER BY 子句指定的欄位組合。主要是決定 Data Part 中資料的物理排序方式，並影響查詢範圍裁剪的效率。
